# AI 交互与生成准则 (AI Interaction & Generation Guidelines)

## 1. 代码修改与生成原则

- **只做最小必要修改**: 严格聚焦于当前用户的需求。除非用户明确要求，否则**绝对不要**去修改、重构或格式化与当前任务无关的模块、代码行或配置文件。
- **代码完整性**: 生成或写入的代码片段必须是完整的、可运行的。**严禁**使用 `// ... existing code ...` 或 `/* 此处省略 */` 等占位符，除非你当前调用的代码编辑工具明确支持基于代码块替换（Search/Replace）的操作机制。
- **遵循现有规范**: 编写代码前必须先审视周围的代码风格、命名规范以及项目整体架构（如 uni-app x 的 uvue/uts 解耦规范），保持高度一致性。

## 2. 模块交接文档规范 (Mandatory - 强制执行)

为了保障多轮 AI 会话或跨 Session 时的上下文无缝衔接，**每次完成特定业务模块的开发、重构或复杂 Debug 之后，AI 必须强制执行以下交接流程**：

- **生成/更新交接文档**: **必须**在项目的 `docs/` 目录下（或该模块专属的子目录中，如 `docs/auth/handover.md`、`docs/map/handover.md`）生成或更新一份该模块的**交接文档**。
- **任务状态同步更新**: 交付任何功能模块时，**必须同步在对应的文档或任务清单（如 `docs/AI-Coding-TodoList.md`）中，将该模块对应的任务标记为已完成（打勾 `[x]`）**。并在旁边注明**完成日期、版本号及负责 AI 助手的标识信息**（例如：`✅ [2026-03-31 - v1.0.0 - by Trae AI]`），以确保交付可追溯、状态可验证。
- **交接文档标准结构**: 文档必须包含以下核心章节，以便后续 AI 快速恢复上下文：
  1. **模块概述与核心实现思路**: 简明扼要地说明该模块的功能边界、业务逻辑流转以及核心技术实现（例如：状态是如何管理的、数据是如何获取的）。
  2. **关键的依赖关系**: 明确列出该模块强依赖的内部文件（如特定的 `services`、`utils`、`types` 或 `store`）以及外部组件，勾勒出清晰的数据流向。
  3. **未解决的问题 (Todo/Known Issues)**: 客观记录当前模块中已知但尚未解决的问题、性能隐患、或者是临时妥协的方案（如硬编码的 Mock 数据、尚未处理的异常边界等）。
  4. **下一步开发建议**: 为后续接手的 AI 或开发者提供明确的 Action Item，例如“下一步应将 mock 接口替换为真实 API 联调”、“需要补充全局鉴权拦截器”等。

## 3. 验证与交互沟通

- **交付前自我验证**: 在宣布模块完成并生成交接文档前，AI 需通过静态分析或常识推理，确保已修改的代码没有引入语法错误、未破坏原有的依赖关系。
- **总结与指引**: 在向用户交付最终结果时，应简要总结修改的核心文件，并附上新生成/更新的 `handover.md` 文档链接，引导用户查看或基于此进入下一阶段任务。
